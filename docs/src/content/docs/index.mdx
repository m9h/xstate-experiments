## Why statecharts?

Every behavioral experiment follows the same pattern: present a stimulus, wait for a response, evaluate it, record data, decide what comes next. This is a state machine. It has always been a state machine. jsPsych2 makes that explicit.

## The architecture at a glance

```
experiment machine
  |-- instructions
  |-- blocks (sequential)
  |     |-- block machine (invoked actor)
  |           |-- trial machine (invoked actor)
  |           |     |-- fixation
  |           |     |-- stimulus (entry: record onset)
  |           |     |-- response (wait for event)
  |           |     |-- evaluation (guards: correct? retry?)
  |           |     +-- feedback
  |           +-- checkNext (guard: hasMoreTrials)
  |-- debriefing
  +-- complete (final state -> full dataset)
```

Data flows **upward** through actor outputs. Each trial emits structured results when it reaches its final state. Blocks collect trial outputs. The experiment collects block outputs. No global data store, no side-channel mutation.

## Built on XState v5

jsPsych2 uses the full statechart formalism (Harel, 1987) plus the actor model (Hewitt, 1973) as implemented by [XState v5](https://stately.ai/docs/xstate). This gives you hierarchical states, parallel regions, guards, delayed transitions, and actor composition out of the box.

The runtime dependency is `xstate` (~15KB gzipped). Rendering is a separate concern -- subscribe to state changes and render however you want: vanilla DOM, React, Lit web components, or PixiJS for timing-critical paradigms.
